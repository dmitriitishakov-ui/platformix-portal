# Прототип корпоративного портала

Тестовое задание для стажёра (отдел автоматизации).
Прототип внутреннего корпоративного портала: список документов, поиск, карточка документа, создание документа с авторизацией.

## Стек

- Python + Flask — backend и REST API
- SQLite — хранение документов и пользователей
- JWT (PyJWT) — авторизация
- Swagger / OpenAPI (flasgger) — документация API
- HTML / CSS / JavaScript (без фреймворков) — frontend

## Возможности

- Главная страница со списком документов
- Поиск по документам (по названию, тексту и категории)
- Карточка отдельного документа
- Создание нового документа (требует входа)
- Авторизация по логину и паролю с выдачей JWT-токена
- Интерактивная документация API через Swagger

## API

| Метод | Endpoint           | Назначение            | Авторизация |
|-------|--------------------|-----------------------|-------------|
| POST  | /login             | Получить JWT-токен    | —           |
| GET   | /documents         | Список документов     | —           |
| GET   | /documents?q=текст | Поиск по документам   | —           |
| GET   | /documents/{id}    | Карточка документа    | —           |
| POST  | /documents         | Создание документа    | требуется   |

Документация API доступна после запуска по адресу http://localhost:8080/apidocs

Тестовый пользователь: `admin` / `admin123`

При первом запуске база создаётся автоматически и наполняется тестовыми данными.

## Запуск

```bash
pip install -r requirements.txt
python app.py
```

Открыть в браузере: http://localhost:8080

## Запуск через Docker

```bash
docker compose up
```

Открыть: http://localhost:8080

## AI-first подход

Проект разрабатывался с активным использованием ИИ-инструментов (Claude). Ниже честное описание того, как именно ИИ применялся.

### Какие задачи решались через ИИ

- Проектирование структуры проекта и API под требования задания
- Написание Flask-приложения с эндпоинтами и JWT-авторизацией
- Подключение Swagger для документации API
- Вёрстка и логика frontend на чистом JavaScript
- Составление Dockerfile, docker-compose и CI-конфигурации
- Объяснение незнакомых частей кода простым языком

### Какие промпты использовались

- «Сделай Flask-приложение для прототипа корпоративного портала: эндпоинты GET /documents, GET /documents/{id}, POST /documents, хранение в SQLite, mock-данные»
- «Сделай frontend на чистом HTML/CSS/JS: список документов, поиск, карточка, навигация — без фреймворков»
- «Добавь JWT-авторизацию: POST /login выдаёт токен, создание документа защищено токеном»
- «Подключи Swagger, чтобы документация API открывалась на /apidocs»
- «Объясни простыми словами, что делает каждый эндпоинт»

### Где ИИ помог

- Быстро собрал рабочий каркас проекта
- Подсказал структуру REST API и подключение JWT
- Помог разобраться, как Flask отдаёт JSON и как frontend получает его через Fetch API

### Где ИИ ошибался

- Первый вариант поиска использовал SQL-оператор `LIKE` напрямую в базе. На русских словах в разном регистре («Регламент» против «регламент») поиск не находил совпадений — SQLite по умолчанию не приводит кириллицу к одному регистру. Пришлось переделать: поиск теперь выполняется на стороне Python с приведением к нижнему регистру. Это пример того, что сгенерированный код нужно проверять запуском, а не принимать на веру.

### Вывод

ИИ-инструменты ускорили разработку, но контроль результата оставался за мной: каждый кусок кода проверялся запуском, а ошибки (как с поиском) исправлялись вручную после тестирования.

## Принятые решения

- **Flask** — лёгкий фреймворк, которого достаточно для прототипа. Быстро запускается, понятен для чтения.
- **SQLite** — встроенная в Python база, не требует отдельного сервера. Для прототипа достаточно; при росте меняется на PostgreSQL без изменения логики.
- **JWT для авторизации** — стандартный способ: после входа клиент получает токен и прикладывает его к защищённым запросам. Сервер не хранит сессии.
- **Frontend без фреймворка** — задача простая, React или Vue были бы избыточны. Чистый JS с Fetch API решает всё и оставляет код прозрачным.
- **Swagger** — автоматическая документация API, по которой видно все эндпоинты и можно их протестировать прямо из браузера.
- **mock-данные при первом запуске** — чтобы проверяющий сразу увидел рабочий портал.
