# Прототип корпоративного портала

Тестовое задание для стажёра (отдел автоматизации).
Часть 1 — мини-прототип внутреннего корпоративного портала: список документов, поиск, карточка документа, создание документа.

## Стек

- Python + Flask — backend и REST API
- SQLite — хранение документов
- HTML / CSS / JavaScript (без фреймворков) — frontend

## Возможности

- Главная страница со списком документов
- Поиск по документам (по названию, тексту и категории)
- Карточка отдельного документа
- Создание нового документа через форму
- Навигация между разделами

## Структура проекта

```
platformix_portal/
  app.py                — Flask-приложение и API
  requirements.txt      — зависимости
  templates/
    index.html          — весь frontend (HTML + CSS + JS)
```

## API

| Метод | Endpoint           | Назначение            |
|-------|--------------------|-----------------------|
| GET   | /documents         | Список документов     |
| GET   | /documents?q=текст | Поиск по документам   |
| GET   | /documents/{id}    | Карточка документа    |
| POST  | /documents         | Создание документа    |

При первом запуске база автоматически создаётся и наполняется тестовыми (mock) данными.

## Запуск

```bash
pip install -r requirements.txt
python app.py
```

После запуска открыть в браузере: http://localhost:8080

## Запуск через Docker

```bash
docker compose up
```

После запуска открыть: http://localhost:8080

### Где ИИ помог

- Быстро собрал рабочий каркас проекта, который иначе занял бы намного больше времени
- Подсказал структуру REST API в соответствии с требованиями
- Помог разобраться, как Flask отдаёт данные в формате JSON и как frontend их получает через Fetch API

### Где ИИ ошибался

- Первый вариант поиска использовал SQL-оператор `LIKE` напрямую в базе. На тестовых данных поиск по русским словам в разном регистре («Регламент» против «регламент») не находил совпадений — SQLite по умолчанию не приводит кириллицу к одному регистру. Пришлось переделать: теперь поиск выполняется на стороне Python с приведением к нижнему регистру. Это рабочий пример того, что сгенерированный код нужно проверять, а не принимать на веру.

### Вывод

ИИ-инструменты ускорили разработку, но контроль результата оставался за мной: каждый кусок кода проверялся запуском, а ошибки (как с поиском) исправлялись вручную после тестирования.

## Принятые решения

Краткое объяснение, почему выбран именно такой стек.

- **Flask** — лёгкий веб-фреймворк, которого достаточно для прототипа с несколькими эндпоинтами. Не тянет лишнего, быстро запускается, понятен для чтения.
- **SQLite** — встроенная в Python база, не требует отдельной установки и сервера. Для прототипа этого хватает; при росте проекта меняется на PostgreSQL без изменения логики.
- **Frontend без фреймворка (чистый HTML/CSS/JS)** — задача простая (список, поиск, карточка, форма), React или Vue здесь были бы избыточны. Чистый JS с Fetch API решает всё и оставляет код прозрачным.
- **Разделение на API и страницу** — backend отдаёт данные в JSON, frontend их получает и отрисовывает. Это стандартный подход, при котором тот же API можно потом использовать из мобильного приложения или другого клиента.
- **mock-данные при первом запуске** — чтобы проверяющий сразу увидел рабочий портал с документами, без ручного наполнения базы.
