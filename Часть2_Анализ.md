# Часть 2. Reverse engineering / анализ legacy-системы

## 1. Краткое описание системы

### Что делает система

Система ведёт учёт заказов и платежей по ним. Основная задача — показывать, кто из клиентов недоплатил: по каждому заказу считается сумма всех поступивших платежей, вычитается из суммы заказа, и в отчёт попадают те заказы, где остался долг. По-простому это отчёт о задолженности клиентов (дебиторская задолженность).

### Основные сущности

**Orders (Заказы)** — сами заказы: кто заказал, на какую сумму, в каком статусе, когда создан.

| Поле | Тип | Назначение |
|---|---|---|
| Id | INT, PK | Номер заказа |
| CustomerName | NVARCHAR(100) | Имя клиента |
| Amount | DECIMAL(10,2) | Сумма заказа |
| Status | NVARCHAR(50) | Статус (есть как минимум 'Cancelled') |
| CreatedAt | DATETIME | Дата создания |

**Payments (Платежи)** — оплаты по заказам. Один заказ может иметь несколько платежей (оплата частями).

| Поле | Тип | Назначение |
|---|---|---|
| Id | INT, PK | Номер платежа |
| OrderId | INT | К какому заказу относится |
| PaidAmount | DECIMAL(10,2) | Сумма платежа |
| PaidAt | DATETIME | Дата платежа |

### Какой бизнес-процесс реализован

1. Создаётся заказ с суммой и статусом.
2. Клиент оплачивает заказ — возможно, частями. Каждая оплата это отдельная строка в Payments.
3. Запрос по каждому заказу складывает все платежи и считает долг (сумма заказа минус оплачено).
4. В отчёт попадают только заказы с долгом больше нуля, у которых статус не «Cancelled».

Важная деталь в запросе — `LEFT JOIN`. Он нужен, чтобы в отчёт попадали и заказы, по которым вообще не было ни одного платежа. При обычном соединении такие заказы выпали бы из отчёта — а это как раз клиенты, которые не заплатили ничего, то есть самые крупные должники. `ISNULL(SUM(...), 0)` для таких заказов подставляет 0 вместо «пусто», и весь долг корректно попадает в отчёт.

### Какие риски и проблемы видны

1. **Клиент хранится текстом.** Имя клиента записано строкой в каждый заказ. Один и тот же клиент в разных заказах может быть записан по-разному — «ООО Ромашка», «Ромашка ООО», с опечаткой. Для базы это разные клиенты, поэтому посчитать общий долг одного клиента по всем заказам нельзя. Решение — отдельная таблица клиентов, а в заказе хранить ссылку на неё.

2. **Нет связи между таблицами на уровне базы.** Payments.OrderId никак не привязан к Orders.Id (нет внешнего ключа). Можно случайно записать платёж на несуществующий заказ — и он просто потеряется, нигде не отразившись.

3. **Статус — произвольный текст.** Поле Status это просто строка. Опечатка вида 'Canceled' (с одной L) — и отменённый заказ всё равно попадёт в отчёт о долгах, потому что он не совпадёт с 'Cancelled'. Нужен либо справочник статусов, либо ограничение на допустимые значения.

4. **Переплата не контролируется.** Ничто не мешает сумме платежей превысить сумму заказа. Такой заказ просто не покажется в отчёте (долг отрицательный), и ошибку в данных никто не заметит.

5. **Нет индексов и нет валюты.** На больших объёмах данных запрос будет медленным без индексов на OrderId и Status. Если компания работает с разными валютами — суммы складывать напрямую нельзя, а в схеме валюта вообще не предусмотрена.

## 2. Карточка системы

**Название системы:** Учёт заказов и задолженности клиентов (в коде название не задано — указано по смыслу).

**Назначение:** Учёт заказов и платежей по ним, контроль задолженности. Главный сценарий — формирование списка заказов с неполной оплатой (отчёт о дебиторской задолженности).

**Основные сущности:**
- Заказ (Orders) — кто заказал, на какую сумму, статус, дата.
- Платёж (Payments) — оплата по заказу, возможно частичная.

**Пользователи (предположительно):**
- Бухгалтерия и финансовый отдел — главный потребитель отчёта о долгах.
- Менеджеры по продажам — контроль оплат по своим клиентам.
- Руководство — общая картина задолженности.

**Возможные интеграции:**
- Бухгалтерская система (например 1С) — обмен данными о платежах.
- CRM — данные о клиентах и заказах.
- Банк-клиент — автоматическая загрузка поступивших платежей.
- Уведомления (email / мессенджеры) — напоминания о просроченных долгах.

**Основные процессы:**
1. Регистрация заказа.
2. Регистрация платежей, в том числе частичных.
3. Расчёт долга по каждому заказу.
4. Исключение отменённых заказов из отчётности.

**Потенциальные риски:**
- Целостность данных не защищена базой (нет внешнего ключа, статусы не ограничены).
- Дубли клиентов из-за хранения имени текстом — нельзя посчитать общий долг клиента.
- Переплаты не отслеживаются.
- Производительность на больших объёмах (нет индексов).
- Бизнес-логика «зашита» в один SQL-запрос: при изменении правил (новые статусы, другая логика долга) придётся вручную искать и править все подобные запросы.

**Рекомендации при доработке / миграции:**
- Вынести клиентов в отдельную таблицу, в заказе хранить ссылку.
- Добавить внешний ключ Payments → Orders.
- Заменить текстовые статусы справочником или ограничением.
- Добавить индексы на поля соединения и фильтрации.
